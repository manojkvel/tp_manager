# Incident Postmortem Synthesizer

Analyze your project's incident history — from bug tickets, hotfix commits, deployment rollbacks, and any existing postmortem notes — to synthesize patterns across incidents. Answers: which services fail most? What root causes recur? Do incidents cluster after certain types of changes? Produces a structured postmortem report with actionable patterns, not just a retelling.

## Step 1 — Gather Incident Data Sources

Collect incident signals from multiple sources:

### Git History — Hotfix & Rollback Detection
```bash
# Hotfix branches
git branch -r --list '*hotfix*' '*fix*' '*patch*' '*revert*' 2>/dev/null | head -20

# Revert commits (explicit rollbacks)
git log --all --oneline --grep='revert' --since='6 months ago' | head -20
git log --all --oneline --grep='rollback' --since='6 months ago' | head -10

# Hotfix commits (emergency fixes)
git log --all --oneline --grep='hotfix' --since='6 months ago' | head -20
git log --all --oneline --grep='urgent' --since='6 months ago' | head -10
git log --all --oneline --grep='emergency' --since='6 months ago' | head -10

# Commits outside business hours (potential incident response)
git log --format='%H %ai %s' --since='6 months ago' | head -100
```

### Existing Postmortem Documents
```
Glob: **/postmortem*, **/incident*, **/post-mortem*, docs/incidents/*, docs/postmortems/*
Glob: **/RCA*, **/root-cause*, reports/incident-*
```

### Bug/Incident Reports in Repository
```
Glob: .github/ISSUE_TEMPLATE/bug*, .azuredevops/bug*
Grep: "severity.*critical", "priority.*1", "P0", "P1", "SEV-1", "SEV-2"
```

### Deployment Artifacts
```
Glob: **/deploy*, **/release-notes*, CHANGELOG.md
```
Look for entries mentioning fixes, reverts, or incidents.

If `$ARGUMENTS` is `recent`, focus on the last 3 months. If a date range is provided, filter accordingly. If `full`, analyze all available history.

## Step 2 — Build Incident Timeline

For each detected incident, construct a record:

```
Incident ID: <auto-generated or from ticket>
Date: <when it happened>
Severity: <P0/P1/P2/P3 — inferred from commit urgency, branch type, or explicit tags>
Duration: <time from incident commit to fix commit, if detectable>
Affected Area: <files/modules involved>
Root Cause Category: <see categories below>
Detection Method: <how it was found — monitoring, user report, deploy failure>
Resolution: <what was done to fix it>
Commits: <list of related commit hashes>
```

### Root Cause Categories

Classify each incident into one of these categories:
1. **Code Defect** — Logic error, off-by-one, null pointer, race condition
2. **Configuration Error** — Wrong env var, bad config value, missing secret
3. **Dependency Failure** — Third-party service down, breaking dependency update
4. **Infrastructure Issue** — Server crash, resource exhaustion, network failure
5. **Data Issue** — Bad migration, corrupt data, missing records
6. **Security Incident** — Vulnerability exploited, unauthorized access
7. **Deployment Error** — Bad deploy, missing artifact, wrong version
8. **Performance Degradation** — Slow queries, memory leak, CPU spike
9. **Integration Failure** — API contract broken, schema mismatch
10. **Human Error** — Manual mistake, wrong command, access misconfiguration

## Step 3 — Analyze Patterns

### Frequency Analysis
- **Incidents per month** — Is the rate increasing, decreasing, or stable?
- **Incidents by severity** — Distribution of P0/P1/P2/P3
- **Incidents by root cause** — Which categories appear most?

### Temporal Patterns
- **Day of week** — Do incidents cluster on certain days? (e.g., Friday deploys)
- **Time of day** — Are incidents happening outside business hours?
- **Sprint phase** — Do incidents cluster after major releases?
- **Seasonal trends** — Any spikes during high-traffic periods?

### Hotspot Analysis
```bash
# Files most involved in incident-related commits
git log --all --oneline --grep='fix\|hotfix\|revert\|bug\|incident' --since='6 months ago' --format=format: --name-only | sort | uniq -c | sort -rn | head -20
```

- **Files with most fixes** — These are reliability hotspots
- **Modules with most incidents** — These need architecture attention
- **Authors handling most incidents** — Potential knowledge silos or on-call burden

### Root Cause Recurrence
Group incidents by root cause category:
- Are the same root causes recurring? (e.g., repeated config errors)
- Which categories have NO incidents? (positive signal)
- Is there a trend? (e.g., more dependency failures lately)

### Change Correlation
```bash
# Commits in the 48 hours before each incident
# For each incident date, check what was deployed
git log --since='<incident-date - 2 days>' --until='<incident-date>' --oneline | head -10
```

Look for patterns:
- Do incidents follow large PRs (> 500 lines changed)?
- Do incidents follow changes to specific modules?
- Do incidents follow dependency updates?

## Step 4 — Calculate Metrics

### Key Metrics

| Metric | Formula | Value |
|--------|---------|-------|
| **MTTR** (Mean Time to Resolve) | avg(resolution_date - incident_date) | N hours |
| **MTBF** (Mean Time Between Failures) | avg(time between consecutive incidents) | N days |
| **Incident Rate** | incidents / month | N/month |
| **Recurrence Rate** | repeated_root_causes / total_incidents | N% |
| **Severity Distribution** | P0: N%, P1: N%, P2: N%, P3: N% | |
| **Fix Revert Rate** | fixes_that_were_reverted / total_fixes | N% |

### Trend Assessment

For each metric, compare the last 3 months to the previous 3 months:
- ↑ **Improving** — metric moving in healthy direction
- ↓ **Degrading** — metric moving in unhealthy direction
- → **Stable** — no significant change

## Step 5 — Generate Recommendations

Based on pattern analysis, generate actionable recommendations:

### For Recurring Root Causes
If the same root cause appears 3+ times:
```
[RECURRING] <Root Cause Category> — <N occurrences in last 6 months>
  Pattern: <what's happening>
  Prevention: <specific engineering action>
  Investment: <small/medium/large effort>
  Expected Impact: <how many incidents this would prevent>
```

### For Hotspot Files/Modules
```
[HOTSPOT] <file/module path> — <N incidents>
  Suggestion: <refactor, add tests, add monitoring, split module>
  Related incidents: <list>
```

### For Process Gaps
```
[PROCESS] <gap description>
  Evidence: <what data shows this>
  Recommendation: <process change>
```

### Prioritized Action Items
Rank all recommendations by: (frequency × severity × effort_inverse)

## Step 6 — Format Output

### Incident Postmortem Synthesis

```
Analysis Period: <start date> — <end date>
Total Incidents Detected: N
Data Sources: git history, postmortem docs, deployment artifacts
```

### Executive Summary
2-3 sentences: overall incident health, top concern, top improvement.

### Metrics Dashboard

| Metric | Current (3mo) | Previous (3mo) | Trend |
|--------|---------------|-----------------|-------|
| Incident Rate | N/mo | N/mo | ↑/↓/→ |
| MTTR | N hrs | N hrs | ↑/↓/→ |
| MTBF | N days | N days | ↑/↓/→ |
| Recurrence Rate | N% | N% | ↑/↓/→ |
| P0/P1 Count | N | N | ↑/↓/→ |

### Incident Timeline
Chronological list of all detected incidents with severity, root cause, and resolution.

### Pattern Analysis
- Root cause distribution chart (text-based)
- Temporal patterns
- Hotspot files and modules
- Change correlation findings

### Recommendations (Prioritized)
Top 5 actionable items with expected impact.

### Appendix — Full Incident Log
Table of all incidents with complete details.

## Step 7 — Save Report

Save the complete synthesis to a persistent file.

1. Create the `reports/` directory if it doesn't exist: `mkdir -p reports`
2. Get today's date: `date +%Y-%m-%d` and capture as `$DATE`
3. Determine the scope label:
   - `full`, `recent`, or the date range provided
4. Save to: `reports/incident-postmortem-<scope>-<DATE>.md`
   - Include a YAML front-matter header with: `date`, `scope`, `total_incidents`, `mttr_hours`, `mtbf_days`, `recurrence_rate_pct`, `p0_count`, `p1_count`
5. Print the file path so the user knows where to find it

**Naming examples:**
- `reports/incident-postmortem-full-2025-06-15.md`
- `reports/incident-postmortem-recent-2025-06-15.md`

**Tip:** Run monthly after sprint retrospectives. Feed the output into `/report-trends` to track incident health over time. Pair with `/release-readiness-checker` to correlate incidents with release quality.
